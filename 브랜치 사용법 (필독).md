# 🌿 Git 브랜치 사용 가이드

> FitPico 프로젝트 팀원을 위한 브랜치 운영 매뉴얼

---

## 브랜치 구조 한눈에 보기

```
main                        ← 최종 배포 (Day 4에만 머지)
  └── develop               ← 통합 브랜치 (팀원들이 여기로 PR)
        ├── feature/sensor-hcsr04     (팀원 A)
        ├── feature/display-lcd       (팀원 B)
        └── feature/iot-dashboard     (팀원 C)
```

---

## Step 1 · 프로젝트 시작 — 레포 클론 & 초기 세팅

```bash
# 1. 레포 클론
git clone https://github.com/팀명/fitpico.git
cd fitpico

# 2. 현재 브랜치 확인
git branch
#  * main  ← 현재 위치

# 3. develop 브랜치로 이동
git checkout develop
# 없으면 새로 만들기
git checkout -b develop
```

> **`-b` 옵션이란?** 브랜치를 새로 만들면서 동시에 이동하는 단축키.  
> `git branch develop` + `git checkout develop`을 한 번에 합친 것.

---

## Step 2 · 기능 개발 시작 — 내 feature 브랜치 만들기

| 팀원 | 명령어 |
|------|--------|
| 팀원 A | `git checkout -b feature/sensor-hcsr04` |
| 팀원 B | `git checkout -b feature/display-lcd` |
| 팀원 C | `git checkout -b feature/iot-dashboard` |

```bash
# develop에서 분기해서 내 브랜치 생성
git checkout develop
git checkout -b feature/sensor-hcsr04

# 지금 어디 있는지 확인
git branch
#   develop
# * feature/sensor-hcsr04  ← 여기!
```

> ⚠️ **주의:** feature 브랜치는 반드시 `develop`에서 만들어야 합니다.  
> `main`에서 만들면 나중에 머지할 때 꼬입니다.

---

## Step 3 · 코드 작성 — 작업하고 커밋하기

```bash
# 코드 작성 후...

# 1. 변경 파일 확인
git status

# 2. 파일 스테이징 (커밋 준비)
git add sensor/hcsr04.py
# 전체 추가하려면
git add .

# 3. 커밋
git commit -m "feat: HC-SR04 거리 측정 함수 추가"

# 4. 원격(GitHub)에 올리기
git push origin feature/sensor-hcsr04
```

### 커밋 메시지 컨벤션

```
feat:     새 기능 추가
fix:      버그 수정
docs:     문서 수정
refactor: 코드 리팩터링
test:     테스트 코드
```

예시:
```
feat: HC-SR04 거리 측정 함수 추가
fix: PIR 센서 오감지 필터링
docs: README 회로도 추가
```

> 💡 **커밋은 자주 작게!** 하루에 5~10개씩 올려두면 충돌 날 때 되돌리기 쉬워요.

---

## Step 4 · 팀원 코드 반영 — develop 최신 코드 내려받기

```bash
# 팀원이 develop에 머지했을 때 내 브랜치에 반영하기

# 1. develop 최신화
git checkout develop
git pull origin develop

# 2. 내 브랜치로 돌아와서 머지
git checkout feature/sensor-hcsr04
git merge develop

# 충돌(conflict)이 없으면 완료!
# 충돌 났을 때 → 파일 열어서 수동으로 해결 후
git add . && git commit -m "fix: merge conflict 해결"
```

> ⚠️ **매일 아침 필수!** 작업 시작 전 develop pull 먼저 받는 습관을 들이세요.  
> 안 하면 충돌이 눈덩이처럼 쌓입니다.

---

## Step 5 · 코드 리뷰 — Pull Request 올리기

```bash
# PR 올리기 전에 최신 develop 반영 먼저!
git merge develop
git push origin feature/sensor-hcsr04

# 그 다음 GitHub 웹에서
# feature/sensor-hcsr04 → develop 으로 PR 생성
```

### PR 규칙

- **제목 형식:** `[feat] HC-SR04 초음파 센서 거리 측정 구현`
- **리뷰어:** 팀원 1명 이상 지정 → 승인 후 머지
- **머지 방식:** `Squash and merge` 선택 → 커밋 이력이 깔끔하게 정리됨

---

## Step 6 · 마무리 — 머지 후 브랜치 정리

```bash
# PR 머지 완료 후 브랜치 삭제

# 로컬 브랜치 삭제
git branch -d feature/sensor-hcsr04

# 원격 브랜치 삭제 (GitHub에서 자동 삭제 설정 가능)
git push origin --delete feature/sensor-hcsr04

# 다음 기능 개발 시작 → develop에서 새 브랜치
git checkout develop
git pull origin develop
git checkout -b feature/sensor-pir
```

> 💡 **GitHub 설정 팁:** Settings → General → "Automatically delete head branches" 켜두면  
> PR 머지 시 원격 브랜치가 자동 삭제됩니다.

---

## 전체 흐름 요약

```
develop pull
    ↓
feature 브랜치 생성
    ↓
작업 / 커밋 / push  (매일 반복)
    ↓
develop pull → 내 브랜치에 merge  (매일 아침)
    ↓
PR 생성 → 리뷰 → 머지
    ↓
브랜치 삭제
```

## 자주 쓰는 명령어 모음

```bash
git branch                        # 브랜치 목록 확인
git branch -a                     # 원격 브랜치 포함 전체 확인
git checkout 브랜치명              # 브랜치 이동
git checkout -b 브랜치명           # 브랜치 생성 + 이동
git status                        # 변경 파일 확인
git add .                         # 전체 스테이징
git commit -m "메시지"             # 커밋
git push origin 브랜치명           # 원격에 push
git pull origin 브랜치명           # 원격에서 pull
git merge 브랜치명                 # 브랜치 머지
git log --oneline --graph         # 브랜치 그래프 확인
```

## 절대 하면 안 되는 것

```bash
# ❌ main에 직접 push 금지
git push origin main

# ❌ develop에서 직접 작업 금지 (반드시 feature 브랜치에서)
git commit -m "..." # develop 브랜치에서 커밋 X

# ❌ secrets.py 커밋 금지 (WiFi 비밀번호 등 민감 정보)
git add secrets.py  # 절대 금지
```
